1. Vue d'ensemble

BJHUNT est une plateforme SaaS de cybersécurité offensive et défensive, entièrement pilotée par un modèle d'intelligence artificielle propriétaire (BJHUNT AI 1.0.0). L'utilisateur interagit via une interface de chat en temps réel : il décrit sa demande en langage naturel, et le système multi-agents sélectionne automatiquement l'agent le plus adapté, exécute des outils de sécurité (nmap, nuclei, nikto, etc.) dans un conteneur Kali Linux isolé, et produit une réponse structurée avec analyses, scores CVSS et recommandations.

Chaque message est chiffré de bout en bout (AES-256-GCM) avec des clés uniques par utilisateur. Les outils sont exécutés dans un sandbox Docker avec validation de cibles, RBAC et journalisation complète. Un système IPS (Intrusion Prevention System) intégré protège l'API en temps réel avec détection brute force, inspection des payloads et blocage IP automatique. L'ensemble fonctionne sans GPU (inférence CPU uniquement) sur un VPS de 31 Go de RAM.

3. Concepts clés

Chaque technologie et concept utilisé dans la plateforme est défini ci-dessous. Rien n'est laissé implicite.

Traefik

Reverse proxy et load balancer moderne, conçu nativement pour Docker. Intercepte toutes les requêtes HTTPS sur api.bjhunt.com, termine la connexion TLS (certificats Let's Encrypt renouvelés automatiquement), et route le trafic vers le conteneur bjhunt-backend sur le port 3001. Gère les headers de sécurité HTTP (HSTS, X-Frame-Options). Configuration dynamique dans /etc/dokploy/traefik/dynamic/bjhunt-api.yml.

Dokploy

Interface web self-hosted (port 3000) de gestion de déploiements Docker. Permet de visualiser, démarrer, arrêter et reconstruire les conteneurs depuis un navigateur. Orchestre le réseau dokploy-network partagé par tous les conteneurs (backend, Ollama, Valkey, Kali MCP, PostgreSQL).

Hono

Framework web ultra-léger pour TypeScript, optimisé pour les edge runtimes (Bun, Deno, Cloudflare Workers). Framework HTTP principal du backend. Fournit routage, middlewares (CORS, rate limiting, headers sécurisés, logging), validation Zod et streaming SSE.

Bun

Runtime JavaScript/TypeScript tout-en-un (exécution, bundling, gestionnaire de paquets). Remplace Node.js pour des performances supérieures. Le backend tourne sur Bun 1.3.9 dans un conteneur Alpine Linux.

Ollama (v0.15.6)

Serveur d'inférence pour modèles de langage en local. Charge le modèle propriétaire BJHUNT AI 1.0.0 et expose une API compatible OpenAI sur le port 11434. Fonctionne entièrement sur CPU (num_gpu=0). Le modèle est préchargé au démarrage du backend via preloadOllamaModel() pour éliminer la latence cold-start.

BJHUNT AI 1.0.0

Modèle d'intelligence artificielle propriétaire, spécialement conçu et entraîné pour la cybersécurité. Supporte le raisonnement Chain-of-Thought (balises <think>), l'appel d'outils au format JSON natif, et la synthèse de rapports structurés. Optimisé pour l'inférence CPU.

MCP (Model Context Protocol)

Protocole standardisé de communication entre un modèle IA et des outils externes. Le serveur MCP tourne dans un conteneur Kali Linux et expose 14 outils via transport SSE. Le backend se connecte en client MCP, envoie les appels d'outils générés par le LLM, et reçoit les résultats en streaming.

Valkey

Fork open-source de Redis, 100% compatible protocole Redis. Utilisé comme store clé-valeur en mémoire pour le rate limiting distribué, le stockage de sessions et le cache de tokens. Port 6379 sur le réseau Docker interne.

PostgreSQL 16 (auto-hébergé)

Base de données relationnelle PostgreSQL 16.11, auto-hébergée sur le VPS dans un conteneur Docker (dokploy-postgres, port 5432, IP interne 10.0.1.3). Accédée via Drizzle ORM (schéma déclaratif TypeScript dans backend/src/db/schema.ts) et le driver pg Pool pour les requêtes paramétrées. Migrations SQL manuelles dans supabase/migrations/ (répertoire local, non lié au service Supabase). L'authentification et toute la logique métier sont gérées par le backend custom Hono.

Drizzle ORM

ORM TypeScript léger et type-safe pour PostgreSQL. Remplace les requêtes SQL brutes (pg Pool) par un schéma déclaratif avec relations, index, migrations automatiques et requêtes typées. Compatible natif Bun. Schéma dans backend/src/db/schema.ts.

Pino

Logger structuré JSON haute performance pour Node.js/Bun. Remplace console.log. Niveaux de log (debug, info, warn, error), rédaction automatique des secrets, sérialisation d'erreurs, transport pino-pretty en développement. Corrélation par requestId.

IPS (Intrusion Prevention System)

Module de prévention d'intrusion intégré au backend. Détection brute force (5 tentatives/15min → blocage), rate limiting global (120 req/min), inspection des payloads (SQL injection, XSS, path traversal), blocage IP automatique avec expiration. Cache en mémoire + persistance PostgreSQL.

Health Monitor

Surveillance continue de tous les services (PostgreSQL, Ollama, Kali MCP, Valkey). Checks toutes les 60 secondes, détection de dégradation (>2s) et de pannes (3 échecs consécutifs). Création automatique d'alertes critiques. Historique complet en base.

Incident Manager

Système de gestion d'incidents avec cycle de vie complet : détection → création → investigation → mitigation → résolution. Timeline détaillée, auto-escalade des incidents critiques, déduplification, association d'alertes. API CRUD complète.

Security Analytics

Tableau de bord sécurité temps réel. Métriques IPS, incidents, exécutions d'outils, modération, alertes actives. Tendances horaires, top IP bloquées, top outils, uptime services, activité d'audit. Données agrégées sur 24h configurable.

Vercel

Plateforme de déploiement du frontend Next.js. Chaque push sur main déclenche un build automatique (~55s) et un déploiement CDN global. Le domaine bjhunt.com pointe vers Vercel. Les pull requests génèrent des preview deploys.

SSE (Server-Sent Events)

Protocole HTTP unidirectionnel serveur → client pour le streaming temps réel. Utilisé pour streamer les réponses du LLM token par token (pensée, contenu, outils, résultats). Chaque événement est préfixé par son type suivi de données JSON.

JWT (JSON Web Token)

Standard d'authentification. Après login, le serveur émet un JWT (courte durée) et un refresh token (cookie httpOnly, 30 jours). Le JWT est envoyé dans Authorization: Bearer <token>. Auto-refresh transparent sur 401.

RBAC (Role-Based Access Control)

Contrôle d'accès par rôles. Chaque utilisateur a un rôle (user, admin) et des permissions d'outils individuelles (table tool_permissions). Avant chaque exécution d'outil : vérification permission + quota journalier + validation cible. Tout refus est journalisé.

AES-256-GCM

Algorithme de chiffrement symétrique. Chaque utilisateur possède une clé unique de 256 bits, chiffrée par une clé maître. Le mode GCM assure confidentialité et intégrité via un authentication tag de 16 octets.

9. Sécurité

La sécurité est intégrée à chaque couche. Aucun message n'est stocké en clair. Aucun outil n'est exécuté sans validation. Toute action sensible est journalisée.

Authentification

JWT courte durée + refresh cookie httpOnly (30j). Clés API bjhunt_ en SHA-256. Auto-refresh sur 401.

Autorisation RBAC

Rôles user/admin. Permissions outils individuelles. Ressources scopées par user_id.

Validation

Zod sur toutes les entrées. JSON Schema pour outils. Détection injection SQL, XSS, prompt.

IPS

Détection brute force (5/15min). Rate limit global (120/min). Inspection payloads. Blocage IP auto avec expiration.

Chiffrement

AES-256-GCM par utilisateur. Clé maître. IV+AuthTag séparés. Rotation supportée.

Monitoring

Health checks 60s. Incidents auto-détectés. Alertes, escalade, analytics temps réel.

Outils

Chaînage shell bloqué. IP privées rejetées. Infra BJHUNT protégée. Quotas et audit.

Modération

Scoring de confiance dynamique. +0.3 si non fiable (3+ bloqués/24h ou 2+ rapports/7j).

Logging

Pino JSON structuré. Rédaction secrets. Corrélation requestId. Niveaux debug→error.

11. Chiffrement

Clé Maître (env: MASTER_ENCRYPTION_KEY) │ └──[AES-256-GCM]──▶ user_encryption_keys │ encryption_key_encrypted (ciphertext) │ encryption_key_iv (16 octets hex) │ encryption_key_auth_tag (16 octets hex) │ ▼ Clé Par Utilisateur (déchiffrée à l'exécution) │ └──[AES-256-GCM]──▶ llm_messages content_encrypted (ciphertext) encryption_iv (IV + AuthTag)

Rotation de clés supportée : tous les messages sont re-chiffrés atomiquement avec la nouvelle clé. La colonne key_version suit les rotations.

Note : La colonne encryption_key_salt est dépréciée. IV et AuthTag sont désormais stockés dans des colonnes séparées (encryption_key_iv, encryption_key_auth_tag) pour plus de robustesse. Les données legacy sont migrées automatiquement.
